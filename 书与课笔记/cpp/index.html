顺序容器

在一个 vector 或 string 的尾部之外的任何位置，或是一个 deque 的首尾之外的任何位置添加元素，都需要移动元素

对于 vector 或 string 插入可能导致所有迭代器失效，删除导致后续迭代器失效。对于 deque 插入、删除首尾之外的位置导致迭代器失效，首尾删除则不会，首尾插入使迭代器失效，但元素引用仍有效，原因。失效后要利用 insert 和 erase 返回值重新定位迭代器，同时不要保存 end

泛型算法

lambda

  • lambda 将其理解为一个未命名的内联函数，可定义在函数内部，不能使用默认参数
  • lambda 的函数体包含任何单一 return 语句之外的内容，且未指定返回类型，则返回 void
  • lambda 可以使用所在函数作用域中的变量，捕获列表只用于局部非 static 变量
  • 在 lambda 创建时拷贝捕获变量。可以采用引用方式捕获变量，但要注意 lambda 执行时引用指向的局部变量消失的问题
  • 对于值拷贝的捕获变量，添加 mutable，令其可修改。引用拷贝的直接改就行

将 bind 函数看作一个通用的函数适配器，定义在头文件 functional 中

在 bind 中使用引用要用标准库 ref 函数

输入迭代器 只读，不写：单遍扫描，只能递增 istream_iterator
输出迭代器 只写，不读：单遍扫描，只能递增 ostream_iterator
前向迭代器 可读写；多遍扫描，只能递增 forward_list
双向迭代器 可读写；多遍扫描，可递增递减 list

  • multi 关键字可以重复出现

  • set or map

动态内存

内存类型

  • static，生存期整个程序运行时
  • stack，生存期在块运行时
  • heap，生存期由程序控制

使用动态内存的 3 种原因

  • 程序不知道自己需耍使用多少对象 -> 用容器
  • 程序不知道所需对象的准确类型 -> 见 15 章
  • 程序需要在多个对象间共享数据 -> 用 shared_ptr

避免 shared_ptr 浪费内存：将 shared_ptr 存放于容器中，而后不再使用某些元素，此时要用 erase 删除不需要的元素

使用 new 和 delete 导致的 3 种错误，1. 忘记释放，或存放地址的变量被覆盖，或发生异常 2. 多次释放 3. 使用已经释放的内存，比如多个变量指向相同的内存

在 new 之后，delete 前发生异常则内存无法被释放，考虑智能指针

不要使用普通指针继续管理交给智能指针的内存，因为不知道内存何时会被释放，也不要用智能指针的 get 继续让其它智能指针一同管理。具体见 P417 智能指针陷阱

可以利用智能指针管理资源，包括发生异常时也可以正确处理，这时传递给智能指针我们自定义的 deleter 来释放资源，见 P416

unique_ptr 不支持拷贝和赋值，例外是从函数可以返回 unique_ptr。管辖权可以转移

weak_ptr 绑定到 shared_ptr，不递增 shared_ptr 计数，只能使用 lock 获取 shared_ptr（不能使用解引用 *），如果释放返回空的 shared_ptr

char *cp = new char[0] 类似尾后指针，不能解引用

  • 有移动构造，则右值初始化时使用移动构造，左值使用拷贝构造

OOP 的核心是数据抽象、继承、动态绑定

基本原则：每个类负责各自成员的访问权限
基类两种函数

  • 直接继承
  • 需要覆盖的

基类通常都应该定义一个虚析构函数，以确保 delete 基类指针时执行正确的析构函数版木

  • 派生类可以访问基类的公有成员和受保护成员
  • 派生类的作用域嵌套在基类的作用域之内
  • 在类名后添加 final 防止被作为基类
  • 如果虚函数使用默认实参，则基类和派生类中定义的默认实参最好一致
  • 派生类调用基类函数时，如果不使用作用域来回避虚函数机制，则会调用自身导致无限循环
  • 含有纯虚函数的类是抽象基类

模版与泛型编程

编译时能确定类型

  • 模板类型参数，关键字 class、typename。用在返回值、参数、函数体内
  • 非类型参数，表示常量值

类模版，编译器不能推断类型。提供默认实参时，用空的尖括号

类中模版的成员函数不能是虚函数

使用 extern 声明在其它文件实例化的模版，可以避免同一类型被实例化多个。要放在其它使用实例化的代码前。
