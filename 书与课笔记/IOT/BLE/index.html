1. 基本概念

蓝牙 4.0 分为两种

  • Basic Rate (BR, often referred to as BR/EDR for Basic Rate/Enhanced Data Rate)
  • Bluetooth low energy (BLE)

同时支持两种的设备为 Bluetooth Smart Ready，比如手机、笔记本。只支持 BLE 的为 Bluetooth Smart，比如使用电池的设备

蓝牙基本概念

BLE 0-39 个信道，每个2Mhz，37、38、39 广播信道，广播信道用来建立连接，其它数据信道进行数据发送

小米 2015 年对于 WiFi、ZigBee、BLE 的选择

  1. 插电的设备，用WiFi。路由器的覆盖范围更大。WiFi可以做到随时随地的连接人和设备、云和设备或者是设备和设备，甚至不同平台之间的对接都很方便，所以也最普适。
  2. 需要和手机交互的，用BLE。低功耗，和手机的互通方便。
  3. 传感器用ZigBee。只有ZigBee联盟有传感器的标准协议，蓝牙联盟还没有。

48 位 IEEE 地址 6B，NAP、UAP IEEE 分配的制造商标识，LAP 制造商分配

  • NAP 无效地址 2B
  • UAP 高位地址 1B
  • LAP 低位地址 3B

1.1 连接与组网

星型结构。主、从 (Central、 Peripheral)

  1. 设备发现。当从节点发出广告信号（包括设备地址和设备名称之类的附加信息）；主节点收到此广告信号后，向从节点发出扫描请求；当从节点回应扫描时，就完成了设备发现过程。
  2. 连接建立。接着主节点向从节点发出连接请求（包括连接时隙、从节点待机次数、连接超时值），从节点回应连接，就完成了建立连接。
  3. 配对。为了安全起见，一些数据的访问需要认证，它的完成是这样的：一方（可以是主节点，也可以是从节点）向另一方索要 6 位数字的密码，之后，两个节点彼此交换安全密钥用于加密和认证，此过程称为配对。
  4. 绑定。认证的过程比较繁琐，BLE 协议支持两节点保存认证的安全密钥（一般是非易失性存储器中），以便于两节点下次连接后快速认证，这就是绑定技术。

广播结构，单向无连接的

  1. Broadcaster 发送广播
  2. Observer 接收广播

连接参数

  • Connection Interval，连接间隔，链路层维护连接的间隔，7.5ms - 4s
  • Slave Latency，从节点待机次数，最大值使有效连接间隔 (Effective Connection Interval) 不超过 16s
  • Supervision Time-out，连接超时值，100ms - 32s。需大于有效连接间隔
  • Effective Connection Interval = (Connection Interval) × (1 + [Slave Latency])
  • 连接参数设置，关于耗电和报文响应的平衡讨论见 CC2540 Software Developer’s Reference Guide

1.2 分层

  • GAP 设备访问模式和进程，包括设备发现、建立连接、终止连接、初始化安全特性、设备配置
  • GAP 层总在 4 种角色之一。Central, Peripheral, Broadcaster, Observer.
  • GAP 从机可以发送连接参数（连接间隔、从机延迟、超时）更新请求给主机，主机接受或拒绝
  • GATT 已连接蓝牙设备间的数据通信。规定了 profile 结构
  • GATT 层中展示 Characteristic 的设备称为服务器，获取 Characteristic 的是客户端
  • GATT 中的客户端、服务器角色与 GAP 角色相互独立，而且两个设备可以同时既是客户端又是服务器，详见 [A view on Bluetooth Low Energy stack roles](A view on Bluetooth Low Energy stack roles.pdf) 和 ble_roles 表格
  • 下图为 GATT 中的三个要素 Profile、Service、Characteristic 以及他们的层级关系，值得注意的是，Profile 其实是 SIG 蓝牙技术联盟给一些同范畴内的 Service 打包后的集合，比如电池、心率、血压等。Service 和 Characteristic 是比较重要的，Service 可以理解为 PHP 中的“类”，功能对象的集合。Characteristic 可以理解为 PHP 的“函数”，是 GATT 中具体的功能对象，每个 Service 都可以包含一个或多个 Characteristic。
  • GATT Service 包含 Mandatory GAP Service、Mandatory GATT Service、Device Information Service、用户自定义的 simpleGATTProfile Service。详见 CC2540 Software Developer’s Reference Guide 5.4.2
  • GATT Characteristic 包含了 4 个 attributes。Characteristic Declaration、Characteristic Value、Client Characteristic Configuration、Characteristic User Description，分别是声明（读写、通知属性，Characteristic 的 handle 和 UUID），值，通知开关，字符串描述
    • 每个 attribute 包含 3 个 properties。handle 属性表的索引，type UUID，Permissions GATT client 对这个 attribute 的访问权限

  • 分层详细参见 低功耗蓝牙(BLE)协议栈简介
  • 通知与指示的区别在于是否发送确认

3. Peripheral 和 Central 代码分析

Peripheral 发送广告的两种模式，受限模式只保持 30s 可发现状态，不受限模式则没有收到 Central 连接请求时一直发送广告，保持可发现状态。通过修改 DEFAULT_DISCOVERABLE_MODE 来选。

Peripheral 流程

  1. SimpleBLEPeripheral_Init
    • Setup the GAP、GAP Peripheral Role Profile、GAP Characteristics，Set advertising interval，Setup the GAP Bond Manager
    • Initialize GATT attributes，注册 Service
    • Setup the SimpleProfile Characteristic Values
    • Register callback with SimpleGATTprofile
    • Setup a delayed profile startup，osal_set_event SBP_START_DEVICE_EVT
  2. simpleBLEPeripheral_HandleKeys
    • 按键回调，通过发送 KEY_CHANGE 消息
    • gapProfileState 不是 GAPROLE_CONNECTED 则反转广告状态
    • 更新按键的 Characteristic，同时开启通知后会通知 Central
  3. peripheralStateNotificationCB
    • 状态变更回调，调用关系 gapRole_ProcessGAPMsg -> peripheralStateNotificationCB (pGapRoles_AppCGs->pfnStateChange)
    • 打印状态。GAPROLE_STARTED 设置 DEVINFO_SYSTEM_ID
    • 修改 gapProfileState
  4. performPeriodicTask
    • 周期性的通过 SimpleProfile_SetParameter 和 SimpleProfile_GetParameter 将第 3 个 Characteristic 赋给第 4 个 Characteristic
  5. simpleProfileChangeCB
    • Characteristic 改变时回调。SimpleProfile_AddService 中注册，调用关系 simpleProfile_WriteAttrCB -> simpleProfileChangeCB
    • 打印新的 Characteristic 值

Central 流程较复杂，先总结如下

  1. 通过按键开启扫描，选择扫描到的设备建立连接，获得 simpleBLEConnHandle
  2. 进行 GATT 服务发现，获得 Characteristic 句柄
  3. 通过 simpleBLEConnHandle 和 Characteristic 句柄进行 Characteristic 读写操作，同时处理读写回应

Central 详细流程

  1. SimpleBLECentral_Init

    • Setup GAP、Central Profile、Setup the GAP Bond Manager
    • Initialize GATT Client
    • Register to receive incoming ATT Indications/Notifications
    • Initialize GATT attributes
    • Setup a delayed profile startup，osal_set_event SBP_START_DEVICE_EVT

  2. SimpleBLECentral_ProcessEvent

    1. SYS_EVENT_MSG

      1. KEY_CHANGE 处理按键 simpleBLECentral_HandleKeys
        • 上：如果没连接则反转扫描开关
        • 上：如果已连接则执行读写操作
        • 左：选择要连接的设备
        • 中：连接或断开设备
        • 右：更新连接间隔、从机延迟、超时
      2. GATT_MSG_EVENT 处理 GATT 消息 simpleBLECentralProcessGATTMsg。详见下面第 7 条

    2. START_DEVICE_EVT

      注册 GAP 回调函数。注册绑定管理回调函数。详见下面第 3-6 条

    3. START_DISCOVERY_EVT

      通过 SIMPLEPROFILE_SERV_UUID 来发现 Peripheral 设备，simpleBLECentralStartDiscovery。客户端通过 UUID 获得 GATT 数据服务，simpleBLESvcStartHdl = simpleBLESvcEndHdl = simpleBLECharHdl = 0 句柄为属性表中的地址。详见下面第 7 条

  3. GAP 状态变化回调 simpleBLECentralEventCB

    • 当一个设备被发现时产生 GAP_DEVICE_INFO_EVENT，通过 UUID 过滤设备后，将设备添加到发现列表
    • 当设备扫描完成时产生 GAP_DEVICE_DISCOVERY_EVENT，显示扫描到的外设个数，提示用户选择
    • 连接建立 GAP_LINK_ESTABLISHED_EVENT，更新状态，启用 START_DISCOVERY_EVT GATT 服务发现
    • GAP_LINK_TERMINATED_EVENT、GAP_LINK_PARAM_UPDATE_EVENT 猜测为按键的操作的回应处理

  4. RSSI 回调 simpleBLECentralRssiCB，Received Signal Strength Indication 接收信号强度指示，将强度值打印出来

  5. GAP 绑定状态回调 simpleBLECentralPairStateCB，把新的绑定状态打印出来

  6. GAP 密钥回调 simpleBLECentralPasscodeCB，生成和显示密钥

  7. 在 3. 连接建立后启用定时发送 GATT 服务发现 simpleBLECentralStartDiscovery 见 2.3 ，回应处理对应 2.1.2 simpleBLECentralProcessGATTMsg

    • simpleBLECentralProcessGATTMsg 中进行 Characteristic 读写的回应处理和 GATT 发现的回应 simpleBLEGATTDiscoveryEvent
    • simpleBLEGATTDiscoveryEvent 的回应处理分为 Service discovery 和 Characteristic discovery
    • 先处理 Service discovery 的回应，保存 simpleBLESvcStartHdl 和 simpleBLESvcEndHdl 句柄，再发起 Characteristic discovery
    • 处理 Characteristic discovery 回应，保存 simpleBLECharHdl 句柄，打印 “Simple Svc Found”

6. 串口透传

主，client 从，server
串口收蓝牙发 发送写 Characteristic 的请求 修改 Profile 中 SetParameter 函数为发送 Notification
蓝牙收串口发 接收 Notification 类型的 Gatt msg 修改 Profile 中 WriteAttrCB 函数为回调相应处理函数
  • 关于 Characteristic 的最大长度，按照 attWriteReq_t 中 value 最大为 ATT_MTU_SIZE-3，而 MTU 为 23，所以设置为 20。但长度大于 19 调 GATT_ReadUsingCharUUID 会返回错误，因为 attReadByTypeRsp_t 的 dataList 大小减去 2B handle 后为 19，20B 的返回数据装不下。
  • server 发送 Notification，可以用 GATTServApp_ProcessCharCfg，但这样需要 client 开启 Notification 而且只能发送固定长度的数据。因此使用 GATT_Notification 发送，只不过需要获得 handle。
  • server 不论收发都可以不真正写入 Characteristic，而是写到最终要使用的内存中。

7. 配对与绑定

参考 CC2540 Software Developer’s Reference Guide 中 5.3 Gap Bond Manager

Term Description
Pairing The process of exchanging keys
Encryption Data is encrypted after pairing, or re-encryption (a subsequent connection where keys are looked up from nonvolatile memory).
Authentication The pairing process complete with MITM (Man in the Middle) protection.
Bonding Storing the keys in nonvolatile memory to use for the next encryption sequence.
Authorization An additional application level key exchange in addition to authentication
OOB Out of Band. Keys are not exchanged over the air, but rather over some other source such as serial port or NFC. This also provides MITM protection.
MITM Man in the Middle protection. This prevents an attacker from listening to the keys transferred over the air to break the encryption.
Just Works Pairing method where keys are transferred over the air without MITM

分 4 步。绑定是可选的

  1. Pair the keys (exchanging keys through the following methods).
    • Just Works (to send the keys wirelessly)
    • MITM (to use a passcode to create a key)
  2. Encrypt the link with keys from step 1.
  3. Bond the keys (store keys in secure flash [SNV]).
  4. When reconnected, use the keys stored in SNV to encrypt the link.

更详细的信息可以看 TI 网站文档，注意最新协议栈可能有些区别 GAP Bond Manager and LE Secure Connections

  • 清除储存的绑定 GAPBondMgr_SetParameter( GAPBOND_ERASE_ALLBONDS,0, NULL );，方便调试

  • 如上面表格配对后就是 Encryption，使用 MITM 配对后就是 Authentication，在 Authentication 基础上在进一步验证就是 Authorization。实现方法就是在 profile 的 add service 注册 callback 的结构体 gattServiceCBs_t 第 3 个成员就是 Authorization callback。理论上区别是 Authorization 要求设备在 device database 被标记为 trust，而 Authentication 只是识别出设备是谁，见 Encryption-Authentication-Authorization
