1. 进程

1.1 理论

进程里面除了程序还有 系统数据段

  • 进程控制块 pcb 即 task_struct，包含 pid、进程用户、文件描述符表、进程状态、优先级、进程返回值
  • cpu寄存器值。PC
  • 堆栈

进程类型

  • 交互进程，shell启动，前台运行，后台运行时无法读取终端输入，但可以输出
  • 批处理进程，与终端无关，提交到作业队列中运行
  • 守护进程，与终端无关，直到系统结束才关闭，一直在后台

进程状态

PROCESS STATE 状态 什么时候产生
R running or runnable 运行态/就绪态 不一定占用 cpu，可能没有时间片
S interruptible sleep 等待态，可否被信号打断，可中断 等待事件或资源
D uninterruptible sleep，Disk wait 等待态，可否被信号打断，不可中断 等待事件或资源。因为不响应信号，所以没办法被用户结束
T stopped, traced 停止态，运行被中止 比如 gdb 调试时，或者 ctrl + Z
Z zombie 死亡态，僵尸态 已经终止但 pcb 没有被释放，pcb 里面有返回值

1.3 fork

返回值

  • 对父进程返回子进程 pid，父进程没有其它办法获知子进程 pid
  • 对子进程返回 0
  • 出错返回 -1

行为

  • 父子进程谁先执行，不一定，看操作系统调度
  • 子进程从 fork 的下一条语句开始执行，子进程继承了 PC
  • 父子不一定谁先结束
    • 若父进程先结束则子进程成为孤儿，被 init 收养，子进程变成后台
    • 若子进程先结束，父进程如果没有及时回收，子进程变成僵尸进程
  • 子进程复制了父的堆栈、全局变量所有内存，也继承了父进程打开的文件，具体看 man fork

1.4 exit

  • exit 比 _exit 多两步，调用退出处理函数，清理标准 IO 缓冲
  • 返回值只有低 8 位有效

1.5 执行

  • exec 原进程内容被替换掉，让父子执行不同程序。shell 就是这么工作的
    • 以 NULL 结尾
    • execlp 不包含路径，使用 PATH 环境变量
    • execl，list。execv，array
    • execvpe 是最终的系统调用，其它函数都是它的封装
  • system 自动创建子进程，父等待执行完成后才继续执行

1.6 进程回收

子进程结束时由父进程回收

  • pid_t wait(int *status)
    • 成功返回子进程号
    • 子进程没结束则父进程阻塞
    • 哪个子进程先结束先被回收
    • status，具体看 man wait 的 example
      • WIFEXITED、WIFSIGNALED 判断是信号结束还是正常结束
      • WTERMSIG 获取是哪个信号导致结束
      • WEXITSTATUS 获取正常结束的返回值
  • pid_t waitpid(pid_t pid, int *status, int options)
    • 通过 pid 指定回收的对象
    • option 指定回收方式。可以选择不阻塞父进程的的方式
    • waitpid(-1, &status, 0) 与 wait 等价

除了常规的 wait，避免僵尸进程的几种办法

  1. 两次 fork，第一次 fork 的子进程立刻退出，第二次 fork 的子进程退出时会被 init 收养。代码较麻烦，并且破坏了父子关系，不推荐

  2. 忽略 SIGCHLD，但无法获取返回值

  3. 注册 SIGCHLD 信号处理函数，函数中调用 waitpid 获得返回值。为什么这样写

1.7 创建守护进程详解

参考 Daemons

进程组和会话等概念解释

举例子更容易理解 Process Groups and Terminal Signaling

To become a daemon, a program performs the following steps:

  1. Perform a fork(), after which the parent exits and the child continues. (As a consequence, the daemon becomes a child of the init process.) This step is done for two reasons:

    1. Assuming the daemon was started from the command line, the parent’s termination is noticed by the shell, which then displays another shell prompt and leaves the child to continue in the background.
    2. The child process is guaranteed not to be a process group leader, since it inherited its process group ID from its parent and obtained its own unique process ID, which differs from the inher ited process group ID. This is required in order to be able to successfully perform the next step.

  3. The child process calls setsid() to start a new session and free itself of any association with a controlling terminal.

  4. If the daemon never opens any terminal devices thereafter, then we don’t need to worry about the daemon reacquiring a controlling terminal. If the daemon might later open a terminal device, then we must take steps to ensure that the device does not become the controlling terminal. We can do this in two ways:

    1. Specify the O_NOCTTY flag on any open() that may apply to a terminal device.
    2. Alternatively, and more simply, perform a second fork() after the setsid() call, and again have the parent exit and the (grand)child continue. This ensures that the child is not the session leader, and thus, according to the System V conventions for the acquisition of a controlling terminal (which Linux follows), the process can never reacquire a controlling terminal.

  6. Clear the process umask by calling umask() to ensure that when the daemon creates files and directories they have the requested permissions.

  7. Change the process’s current working directory by calling chdir(), typically to the root directory (/). This is necessary because a daemon usually runs until system shutdown; if the daemon’s current working directory is on a file system other than the one containing /, then that file system can’t be unmounted. Alternatively, the daemon can change its working directory to a location where it does its job or a location defined in its configuration file, as long as we know that the file system containing this directory never needs to be unmounted. For example, cron places itself in /var/spool/cron.

  8. Close all open file descriptors that the daemon has inherited from its parent. (A daemon may need to keep certain inherited file descriptors open, so this step is optional, or open to variation.) This is done for a variety of reasons. Since the daemon has lost its controlling terminal and is running in the background, it makes no sense for the daemon to keep file descriptors 0, 1, and 2 open if these refer to the terminal. Furthermore, we can’t unmount any file systems on which the long-lived daemon holds files open. And, as usual, we should close unused open file descriptors because file descriptors are a finite resource.

  9. After having closed file descriptors 0, 1, and 2, a daemon normally opens /dev/null and uses dup2() (or similar) to make all those descriptors refer to this device. This is done for two reasons:

    1. It ensures that if the daemon calls library functions that perform I/O on these descriptors, those functions won’t unexpectedly fail.
    2. It prevents the possibility that the daemon later opens a file using descriptor 1 or 2, which is then written to - and thus corrupted - by a library function that expects to treat these descriptors as standard output and standard error.

2. 线程

  • LWP，编译时加 –lpthread
  • linux 为每个进程创建 task_struct，时间片用完，进程切换时 cache 和 TLB 刷新，开销大，同一进程的线程共享相同的地址空间，可以提高任务切换效率
  • linux 不区分进程、线程，如果访问同一块内存的有多个 task_struct ，那么每个就是一个线程
  • gettid （用法见 man syscall example）返回的 pid_t 也就是 ps -L 中的 LWP，是系统唯一的。pthread_self 是用户态的，只是在进程中唯一
  • 创建 pthread_create，回收 pthread_join 会阻塞，结束 pthread_exit 返回值不能是栈上的内存
  • 主线程可以调用 pthread_exit，线程中调用 exit 或 主线程 return 导致进程结束
  • 可以用 pthread_sigmask 创建一个专门处理某个信号的线程，见 example。同样 pthread_kill 发信号

2.1 线程通信

线程间通信全局变量即可，同步与互斥问题考虑信号量和锁。对于同步问题，考虑哪些操作是不能并行的，串行的话顺序是怎样的，来设计信号量。

POSIX 信号量，有名用于 进程间，无名用于 线程间

POSIX 无名信号量 sem_init、sem_wait、sem_post、sem_destroy

临界资源：一次只允许一个任务(进程、线程)访问的共享资源

临界资源用锁，也可以用信号量 pthread_mutex_init、pthread_mutex_lock、pthread_mutex_unlock、pthread_mutex_destroy

2.2 线程属性

  1. 让线程自己清理，不用别人来 join ，则设置线程属性为 detached，默认创建的线程为 joinable。两种方法改变
    • 线程调用 pthread_detach，但再也无法变回 joinable 了
    • 创建时指定属性 pthread_attr_init、pthread_attr_setdetachstate、pthread_attr_destroy
  2. 线程可以被其它线程被动结束，其它线程调用 pthread_cancel 来结束别人。pthread_setcancelstate 来设置自己是否可以被别人 cancel，默认是可以。pthread_setcanceltype 决定是同步还是异步结束，默认同步，则只有在调用某些称为 cancellation point 的函数（通过 man 7 pthreads 查询）时才会结束，异步结束则立刻结束。然后可以通过 pthread_join 的返回值（PTHREAD_CANCELED）来判断线程是否被 cancel ，前提是线程是 joinable 的。
